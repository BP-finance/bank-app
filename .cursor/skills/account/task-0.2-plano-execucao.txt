    - resolveCurrentAccount(): obtém sessão (useAuthStore.getState().session), chama o mapper,

    - Depende apenas do contrato público de AUTH (leitura de session). Não alterar auth.

2.4 hooks/useCurrentAccount.ts
    - useCurrentAccount(): hook que usa useAuthStore((s) => s.session) para reatividade,
      chama o mapper com a session atual, retorna { currentAccount: CurrentAccount | null }.
    - Consumidores usam currentAccount?.id; não acessam session.user.id para accountId.

2.5 index.ts (barrel)
    - Exportar: CurrentAccount, resolveCurrentAccount, useCurrentAccount.
    - Opcional: exportar o mapper se algum teste precisar; caso contrário manter interno.

2.6 README.md (curto) ou comentário em index.ts
    - Texto curto: este módulo é o owner de currentAccount; fallback atual = sessão;
      futura origem = Get Current Account (Idez); não usar session.user.id como accountId fora daqui.

5. MICROAJUSTES POSSÍVEIS
-------------------------

5.1 Import de AUTH dentro de current-account
    - resolveCurrentAccount e useCurrentAccount precisam da sessão.
    - Opção A: importar useAuthStore de @/src/features/auth (barrel). Recomendado para não
      acoplar ao caminho interno da store.
    - Opção B: importar de path relativo se current-account ficar junto de auth na árvore.
    - Resolução: usar @/src/features/auth para useAuthStore e tipos AuthSession/AuthenticatedUser
      importados do barrel de auth, para manter dependência apenas do contrato público.

5.2 Tipo do mapper
    - sessionCurrentAccountFallback(session: AuthSession | null): CurrentAccount | null.
    - Se session for null, retornar null. Se session.user existir, montar CurrentAccount com
      id: session.user.id, source: "session-fallback", ownerUserId: session.user.id,
      personType: session.user.tipoConta (já é "PF"|"PJ"). Documentar que isso é temporário.

5.3 Hook useCurrentAccount e sessão null
    - Quando session for null, currentAccount deve ser null (não objeto com id "").
    - Assim, callers que fazem currentAccount?.id ?? "" ficam corretos (string vazia quando
      não há conta), alinhado ao comportamento atual de accountId vazio quando não logado.

5.4 cards.tsx e ConfirmScreen importam useAuthStore de store
    - Hoje: import { useAuthStore } from "@/src/features/auth/store/useAuthStore".
    - Após a mudança: podem importar useCurrentAccount de @/src/features/current-account.
    - Se não usarem mais session nesses arquivos, remover import de useAuthStore neles.
    - Padrão do projeto: outros arquivos usam @/src/features/auth (barrel). Não é obrigatório
      padronizar na Task 0.2; apenas não derivar accountId da sessão.

5.5 Barrel de current-account
    - Exportar: type CurrentAccount, resolveCurrentAccount, useCurrentAccount.
    - Não exportar o mapper no barrel se não for necessário fora do módulo (recomendado:
      manter mapper interno para não incentivar uso do fallback fora do módulo).

5.6 README vs comentário
    - Mínimo: comentário em sessionCurrentAccountFallback.mapper.ts e uma linha no index.ts
      ou README.md do módulo. README curto (5–10 linhas) é suficiente para “owner de
      currentAccount; fallback temporário; futura origem Get Current Account”.
